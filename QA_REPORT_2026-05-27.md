- **baseTax invariant across date** — expected and verified. Year tax doesn't depend on within-year close date.

- **Strategy C nearly flat** ($983K Jan → $928K Dec, ~5.6% drop). C anchors first payment to Y+1 Jan 1, so close date affects only the basis-tranche Y0 economics.
- **C wins at every date on the canonical scenario** — confirms the "low basis / high gain / full capital" sweet spot for structured sale.

- **A → C crossover** lives somewhere between **gain ratio 11% and 100%** for Jan 1 close. From a separate single-cell at gain ratio 800% (basis $1M / sale $5M) Jan 1 we saw C wins by ~$240K. From low-gain we saw A wins by ~$3K. The exact tipping point at Jan 1 is around **gain ratio ~80–100%**, where the LT/§1411 tax savings on the larger deferred gain start to outweigh the higher Brooklyn fees.
- **A never beats C for high-gain scenarios**, regardless of date.
- **B's invariance lets it win edge cases** — late-year sales with small absolute net push B to the top by elimination, not by being objectively better.

## Fee deep-dive

### F2 detail — Strategy A entry.cfg carries wrong duration

Calling `unifiedTaxComparison(entry.cfg)` for Strategy A on the canonical scenario:

| Source | Brooklyn fees | Brookhaven fees | Total fees |
|---|---:|---:|---:|
| `entry.metrics` (post-optimizer, displayed on Tab 4) | $65,500 | $53,000 | **$118,500** |
| `unifiedTaxComparison(entry.cfg)` re-call | $101,500 | $53,000 | **$154,500** |
| **Delta** | **$36,000 phantom Brooklyn fees** | $0 | $36,000 |

Inspection: `entry.cfg` has `structuredSaleDurationMonths: 36` even though it represents Strategy A (immediate, no structure). The re-call interprets cfg as a structured sale and accrues 4 years of Brooklyn fees ($47K × ~years).

**Fix needed (engine bot):** Either (a) clear `structuredSaleDurationMonths` to 0 when entry.type === 'A', or (b) add a marker (`entry.cfg.strategyType = 'A'`) and have `unifiedTaxComparison` short-circuit when `strategyType === 'A'`.

### F6 detail — perYear field shape

`cmp.brookhavenSchedule` is undefined; the documented `brookhavenSchedule.perYear` shape from `project-rett.md` ("{ perYear: [{setup, quarterly, total}], total }") doesn't appear in actual returned objects. Per-year Brookhaven fees live on each `cmp.rows[i]`:
- `cmp.rows[i].brookhavenFee` — total per year
- `cmp.rows[i].brookhavenSetupFee` — Y0 setup ($45K typical)
- `cmp.rows[i].brookhavenQuarterlyFee` — $8K/year typical

Per-year Brooklyn fees:
- `cmp.rows[i].fee` — Brooklyn fee for that year

**Verified sums reconcile** for engine-direct Strategy C (`{horizon: 4, structuredDurationMonths: 36}`):
- Per-year Brooklyn: $47K × 4 years = $188K → matches `cmp.totalFees: $188K` ✓
- Per-year Brookhaven: $53K (Y0) + $8K (Y1) + $0 + $0 = $61K → matches `cmp.totalBrookhavenFees: $61K` ✓
- Total: $249K → matches `cmp.totalAllFees: $249K` ✓

**At the engine layer, fees reconcile cleanly.** The fee transparency issue is purely in the `buildInterestedSummary` → `entry.cfg` round-trip (F2).

## F1 — drift warning detail

Captured payload from a single normal-scenario render:

```
[RETT drift] Scenario C row vs dashboard mismatch:
  row:  { tax: $380,269,  fees: $205,583,  net: $972,787 }
  dash: { tax: $1,089,935, fees: $105,013,  net: $363,691 }

```

Source: `js/04-ui/projection-dashboard-render.js:1149` in `_assertRowDashboardConsistency`.  
Tolerance: $1 (basically zero — any optimizer dial-back will trip it).  

**Fix needed:** The check should either be (a) removed entirely (its purpose was a one-time correctness check during the optimizer-dial-back rollout), (b) gated on `entry._opt?.dialBack === false`, or (c) widen tolerance to something semantically meaningful (e.g. $100 + 1% of net). Currently it's pure noise — the warning fires on every render with optimizer-active scenarios, which is the common case.

## F5 — autosave fragility (test-infra issue, but symptomatic)

Setting `window.__rettStrategyInterest = {A:true,B:true,C:true}` outside of the autosave-controlled flow gets **overwritten** by autosave/restore in the next render cycle. Two-cell sequences saw interest reset to `{A:null, B:null, C:null}` between cells, causing `buildInterestedSummary` to return all-zero entries.

Workaround that worked in this audit:
1. Set `window.__rettSuppressAutoSave = true` before any DOM mutations.
2. Re-assert `window.__rettStrategyInterest` after each `await wait()`.

**Engine-bot consideration:** the autosave save/restore cycle should respect in-memory state set externally, OR there should be a documented programmatic API to set interest (e.g. `window.rettSetStrategyInterest(obj)` that writes both memory and storage atomically).
